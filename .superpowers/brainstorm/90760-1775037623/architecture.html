LiveKit Network Architecture

How should LiveKit fit into your existing Docker Compose + Traefik setup?

A

LiveKit behind Traefik (WebSocket signaling only)

Traefik handles WSS signaling on port 443. LiveKit exposes UDP ports 7882 + 50000-60000 directly to the host for WebRTC media (UDP can't go through HTTP reverse proxy).

Pros

  • Single TLS termination at Traefik
  • Consistent with existing pattern (go-api, keycloak)
  • Signaling gets Traefik's security headers + rate limiting

Cons

  • UDP ports still exposed directly (unavoidable for WebRTC media)
  • Traefik config slightly more complex (WebSocket upgrade)
B

LiveKit fully standalone (own ports)

LiveKit manages its own TLS on port 7880 (WSS signaling) + 7882 (TCP) + 50000-60000 (UDP media). No Traefik involvement.

Pros

  • Simpler — LiveKit handles everything itself
  • LiveKit docs assume this setup
  • No Traefik WebSocket config needed

Cons

  • Separate TLS cert management for LiveKit
  • Different port (7880) for signaling — firewall/DNS considerations
  • Breaks pattern of Traefik as single ingress
C

LiveKit behind Traefik + TURN colocated

Same as A, but also run LiveKit's built-in TURN server on port 443 (TCP) as fallback for restrictive networks where UDP is blocked.

Pros

  • Works on corporate networks that block UDP
  • Best connectivity for end-users
  • LiveKit has built-in TURN — no separate Coturn needed

Cons

  • Port 443 conflict — TURN and Traefik can't both use 443
  • Needs a separate IP or subdomain for TURN (e.g., turn.raven.example.com)
  • Most complex setup
