# Brainstorming Ideas Into Designs

Help turn ideas into fully formed designs and specs through natural collaborative dialogue.

Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.

<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.

## Checklist

You MUST create a task for each of these items and complete them in order:

1. **Explore project context** — check files, docs, recent commits
2. **Offer visual companion** (if topic will involve visual questions) — this is its own message, not combined with a clarifying question. See the Visual Companion section below.
3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria
4. **Propose 2-3 approaches** — with trade-offs and your recommendation
5. **Present design** — in sections scaled to their complexity, get user approval after each section
6. **Write design doc** — save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit
7. **Spec review loop** — dispatch spec-document-reviewer subagent with precisely crafted review context (never your session history); fix issues and re-dispatch until approved (max 3 iterations, then surface to human)
8. **User reviews written spec** — ask user to review the spec file before proceeding
9. **Transition to implementation** — invoke writing-plans skill to create implementation plan

## The Process

**Understanding the idea:**

- Check out the current project state first (files, docs, recent commits)
- Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
- If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
- For appropriately-scoped projects, ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible, but open-ended is fine too
- Only one question per message - if a topic needs more exploration, break it into multiple questions
- Focus on understanding: purpose, constraints, success criteria

**Exploring approaches:**

- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation and reasoning
- Lead with your recommended option and explain why

**Presenting the design:**

- Once you believe you understand what you're building, present the design
- Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
- Ask after each section whether it looks right so far
- Cover: architecture, components, data flow, error handling, testing
- Be ready to go back and clarify if something doesn't make sense

**Design for isolation and clarity:**

- Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
- For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
- Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
- Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.

**Working in existing codebases:**

- Explore the current structure before proposing changes. Follow existing patterns.
- Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
- Don't propose unrelated refactoring. Stay focused on what serves the current goal.

**Documentation:**

- Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
  - (User preferences for spec location override this default)
- Use elements-of-style:writing-clearly-and-concisely skill if available
- Commit the design document to git

After writing the spec document:

1. Dispatch spec-document-reviewer subagent (see spec-document-reviewer-prompt.md)
2. If Issues Found: fix, re-dispatch, repeat until Approved
3. If loop exceeds 3 iterations, surface to human for guidance

> "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."

Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.

**Implementation:**

- Invoke the writing-plans skill to create a detailed implementation plan
- Do NOT invoke any other skill. writing-plans is the next step.

## Key Principles

- **One question at a time** - Don't overwhelm with multiple questions
- **Multiple choice preferred** - Easier to answer than open-ended when possible
- **YAGNI ruthlessly** - Remove unnecessary features from all designs
- **Explore alternatives** - Always propose 2-3 approaches before settling
- **Incremental validation** - Present design, get approval before moving on
- **Be flexible** - Go back and clarify when something doesn't make sense

A question about a UI topic is not automatically a visual question. "What does personality mean in this context?" is a conceptual question — use the terminal. "Which wizard layout works better?" is a visual question — use the browser.

## Ingestion Pipeline

### Overview

The ingestion pipeline transforms raw documents and web pages into searchable, retrievable knowledge. Every input — whether a file upload or a URL — follows the same lifecycle: **Submit → Queue → Parse/Scrape → Chunk → Embed → Index → Ready**.

### Supported Input Types

| Category | Formats |
|----------|---------|
| Documents | PDF, DOCX, Markdown (.md) |
| Images | PNG, JPG, TIFF (OCR via LiteParse) |
| Web | Any public URL (rendered and scraped) |

### Pipeline Stages

```
┌──────────────┐    ┌───────┐    ┌──────────────┐    ┌───────┐    ┌───────┐    ┌───────┐    ┌───────┐
│ Upload / URL │───▶│ Queue │───▶│ Parse/Scrape │───▶│ Chunk │───▶│ Embed │───▶│ Index │───▶│ Ready │
└──────────────┘    └───────┘    └──────────────┘    └───────┘    └───────┘    └───────┘    └───────┘
     Go API          Redis         Python Worker      Python       Ext. API    pgvector +
                                                      Worker       (BYOK)      ParadeDB
```

### File Processing Path

1. **Go API** receives the multipart upload, validates file type and size, and writes the file to **object storage** (S3/MinIO). A metadata record is created in PostgreSQL with status `queued`.
2. The API enqueues an ingestion job onto the **Redis job queue** containing the file reference, org ID, and embedding provider config.
3. **Python AI worker** picks up the job via gRPC, downloads the file from object storage, and invokes **LiteParse** (Node.js CLI, Apache 2.0) as a subprocess: `liteparse <file> --output json`. LiteParse handles PDF text extraction, DOCX conversion, image OCR, and Markdown normalization.
4. The extracted text proceeds through chunking → embedding → indexing (see below).

### URL Processing Path

1. **Go API** receives the URL, validates it, and enqueues a scrape job onto **Redis**.
2. **Python AI worker** picks up the job and invokes the **web scraper** (see recommendation below) to fetch and render the page, extracting clean Markdown/text content.
3. The extracted content proceeds through the same chunking → embedding → indexing path as file-based inputs.

### Chunking Strategy

The pipeline uses **recursive text splitting**:

- **Target chunk size:** ~512 tokens
- **Overlap:** 50 tokens between consecutive chunks
- Split hierarchy: paragraph boundaries → sentence boundaries → word boundaries
- Metadata preserved per chunk: source document ID, chunk index, section heading (if available), character offsets

This balances retrieval precision (chunks small enough to be topically coherent) with context retention (overlap prevents information loss at boundaries).

### Multi-Provider Embeddings (BYOK)

Each organization configures its embedding provider at the org level:

| Setting | Example |
|---------|---------|
| `embedding_provider` | `openai`, `cohere`, `voyage`, etc. |
| `embedding_model` | `text-embedding-3-small` |
| `embedding_api_key` | Encrypted, org-scoped |
| `embedding_dimensions` | 1536 |

The Python worker reads the org's embedding config from PostgreSQL and dispatches to the appropriate provider client. A thin adapter interface normalizes the call:

```python
class EmbeddingProvider(Protocol):
    def embed(self, texts: list[str]) -> list[list[float]]: ...
```

**Constraint:** All documents within a single org must use the same model/dimensions, since pgvector columns are fixed-width. Changing providers requires re-indexing.

Chunks are embedded in batches (up to provider rate limits) to minimize API round-trips.

### Indexing

Each chunk is written to PostgreSQL with:

- **pgvector** — stores the embedding vector for similarity search
- **ParadeDB BM25** — indexes the raw chunk text for full-text keyword search

At query time, both indexes are queried and results are merged via **Reciprocal Rank Fusion (RRF)**.

### Error Handling and Retry Strategy

| Failure | Behavior |
|---------|----------|
| Parse failure (corrupt file, unsupported format) | Mark document as `failed`, surface error to user. No retry. |
| Scrape failure (timeout, 4xx/5xx) | Retry up to **3 times** with exponential backoff (5s, 30s, 120s). Then mark `failed`. |
| Embedding API error (rate limit, 5xx) | Retry up to **5 times** with exponential backoff. Job remains in queue. |
| Embedding API auth error (invalid key) | Fail immediately, mark `failed`, notify user of bad API key. |
| Partial failure (some chunks embedded, then error) | Resume from last successful chunk on retry (checkpoint-based). Already-indexed chunks are not re-processed. |

All jobs carry a **maximum TTL of 30 minutes** in the queue. Jobs exceeding this are moved to a dead-letter queue for inspection.

Document status is exposed via the API: `queued → processing → ready | failed`.

### Web Scraper Recommendation: Firecrawl vs Crawl4AI

| Criterion | Firecrawl | Crawl4AI |
|-----------|-----------|----------|
| **License** | AGPL-3.0 | Apache 2.0 |
| **Deployment** | Self-hosted or cloud API | Self-hosted (Python library) |
| **JS rendering** | Yes (built-in) | Yes (Playwright-based) |
| **Output quality** | Excellent — clean Markdown, LLM-optimized | Good — Markdown output, improving rapidly |
| **Crawl modes** | Single page + site-wide crawl + sitemap | Single page + multi-page crawl |
| **Integration effort** | REST API or Python SDK | Native Python, direct function calls |
| **License risk** | AGPL requires open-sourcing derivative work if distributed; self-hosted SaaS use is debated | No license risk |

**Recommendation: Crawl4AI (Apache 2.0).**

# Interaction Layer — Raven Platform

## Phase 1: Embeddable Chatbot (MVP)

### 1.1 Architecture — Web Component

Raven ships a framework-agnostic **Web Component** (`<raven-chat>`) built as a single JS bundle. Site owners embed it with one script tag:

```html
<script src="https://cdn.raven.dev/chat.js"></script>
<raven-chat kb="kb_abc123" api-key="rk_live_..."></raven-chat>
```

The component uses Shadow DOM for style isolation, ensuring no CSS collisions with the host page. It renders a floating launcher button, expandable chat window, and message thread. Appearance (colors, logo, welcome message, position) is configured via attributes or the admin dashboard and fetched at init time from a lightweight `/v1/chatbot/{kb}/config` endpoint.

### 1.2 Authentication

- Each **knowledge base** has one or more **publishable API keys** (`rk_live_...`). These are domain-scoped — the Go API validates the key and checks the `Origin` / `Referer` header against an allow-list configured by the org admin.
- No end-user login is required. The API key identifies the tenant and knowledge base; rate limiting and usage metering are applied per key.
- Admin operations (dashboard, config changes) use a separate **secret key** authenticated via standard Bearer tokens over HTTPS.

### 1.3 Chat API — Streaming via SSE

The chat endpoint uses **Server-Sent Events (SSE)** over `POST /v1/chat/{kb}/completions`:

- SSE is chosen over WebSocket for the MVP because it is simpler to deploy behind CDNs/load balancers, works naturally with HTTP/2, and the interaction pattern is request-response (not bidirectional).
- The client sends a JSON body (`message`, `conversation_id`, optional `metadata`). The server responds with `Content-Type: text/event-stream`, streaming tokens as `data:` frames and a final `[DONE]` sentinel.
- Each event includes a `type` field: `token`, `source` (retrieved document references), `error`, or `done`.

### 1.4 RAG Query Flow

```
User message
  │
  ▼
Go API (auth, rate-limit, load conversation history)
  │  gRPC call
  ▼
Python AI Worker
  ├─ 1. Embed query  (provider-specific embedding model)
  ├─ 2. Hybrid search
  │     ├─ Vector similarity search  (pgvector / Qdrant)
  │     └─ BM25 keyword search       (Tantivy / Elasticsearch)
  ├─ 3. Reciprocal Rank Fusion (RRF) to merge results
  ├─ 4. Rerank top-K (Cohere Rerank or cross-encoder)
  └─ 5. LLM generation (BYOK: Anthropic Claude default, org-selected provider)
         │  stream tokens via gRPC server-streaming
         ▼
Go API  ──► SSE stream ──► Client
```

The Go API opens a **gRPC server-streaming** call to the Python worker. Tokens are forwarded to the client as SSE events with minimal buffering. Source citations are sent as a discrete event before the `done` frame.

### 1.5 Conversation History Management

- Each conversation is identified by a `conversation_id` (UUID), returned on the first response and reused by the client on subsequent messages.
- The Go API stores messages in **PostgreSQL** (`conversations` and `messages` tables), scoped to the knowledge base.
- Before calling the AI worker, the Go API loads the last **N turns** (configurable, default 10) and passes them as context. A sliding window with token-budget awareness is used to stay within LLM context limits.
- Conversations have a configurable **TTL** (default 24 h for anonymous sessions). Admins can enable persistent history if they authenticate end-users via a JWT pass-through.

### 1.6 Admin Dashboard (Vue.js + Tailwind Plus)

The Vue.js SPA provides:

- **Chatbot Configurator** — live preview of widget appearance (theme colors, avatar, welcome text, suggested questions).
- **Test Sandbox** — embedded `<raven-chat>` instance pointed at a staging key so admins can test against their knowledge base before going live.
- **Analytics** — conversation volume, unanswered-question rate, top queries, source-hit frequency.
- **API Key Management** — create/revoke publishable keys, set domain allow-lists and rate limits.

## Authentication & Authorization

### Keycloak as Identity Provider

Raven uses Keycloak for all end-user authentication via **OIDC Authorization Code flow with PKCE**:

1. Client (web app) redirects to Keycloak's `/auth/realms/raven/protocol/openid-connect/auth`.
2. User authenticates (credentials, SSO, or social login configured in Keycloak).
3. Keycloak issues an **ID token** + **access token** (JWT) + **refresh token**.
4. Client sends access token as `Authorization: Bearer <token>` to Go API.

### "reavencloak" Custom SPI

The `reavencloak` SPI (built and maintained by the team) extends Keycloak with:

- **Custom User Federation** — syncs organization membership and roles into Keycloak from Raven's Postgres tables.
- **Custom Token Mapper** — injects Raven-specific claims into JWTs: `org_id`, `org_role`, `kb_permissions[]`.
- **Event Listener** — propagates user lifecycle events (registration, deletion) back to Raven's Go API via internal webhook.

Deployed as a JAR in the `kc-config` volume, loaded at Keycloak startup. Configuration is managed through Keycloak's admin realm settings.

### JWT Validation in Go API

The Go API middleware:

1. Extracts the `Bearer` token from the `Authorization` header.
2. Validates signature against Keycloak's JWKS endpoint (`/realms/raven/protocol/openid-connect/certs`), cached with TTL.
3. Checks `iss`, `aud`, `exp`, `nbf` standard claims.
4. Extracts custom claims (`org_id`, `org_role`) and populates the request context.
5. Rejects with `401` on invalid/expired tokens. Downstream handlers check roles for `403`.

### Role-Based Access Control

Organization-scoped roles, carried in the JWT:

| Role | Permissions |
|---|---|
| **Owner** | Full control: billing, delete org, manage all members |
| **Admin** | Manage knowledge bases, invite/remove members, configure integrations |
| **Member** | Create/edit knowledge bases, query, upload documents |
| **Viewer** | Read-only access to knowledge bases they are granted access to |

Enforcement happens in Go API middleware and per-handler authorization checks. Keycloak realm roles map 1:1 to these, managed via `reavencloak` SPI sync.

### API Key Auth (Embeddable Chatbot)

For public-facing chatbot embeds:

- Organization admins generate **scoped API keys** via the Go API (`POST /api/v1/keys`).
- Each key is tied to a specific **knowledge base ID** and permits only `query` operations.
- Keys are hashed (SHA-256) and stored in Postgres; the plaintext is shown once at creation.
- Go API checks `X-API-Key` header, resolves the associated org/KB scope, and enforces rate limits via Redis.
- No user identity — requests are anonymous, scoped, and read-only.

### Strapi Authentication

Strapi runs with its **own admin auth** for CMS administrators (separate credentials, not Keycloak). For API-level access:

- Strapi's REST/GraphQL API is consumed **only by the Go API internally** (not exposed to end users directly for data mutations).
- Go API authenticates to Strapi using a **service account API token** stored in `.env.secrets`.
- End users never interact with Strapi auth. Content permissions are enforced at the Go API layer based on Keycloak JWT claims.

## 2. Service Breakdown

### Go API Server (Gin or FastHTTP)

- **Role:** Primary API gateway for all client requests. Handles routing, tenant resolution, authorization enforcement, and orchestration.
- **Responsibilities:**
  - REST API for CRUD operations (orgs, users, documents, knowledge bases)
  - JWT validation against Keycloak (OIDC token introspection / JWKS verification)
  - Multi-tenant middleware: extracts tenant context from JWT claims or request headers; scopes all DB queries via `org_id`
  - Enqueues async jobs (document processing) into Redis
  - Delegates AI workloads to the Python worker via gRPC
  - Direct PostgreSQL access for transactional data

### Python AI Worker

- **Role:** Dedicated service for all AI/ML workloads. Exposes a gRPC server consumed exclusively by the Go API server.
- **Responsibilities:**
  - Document parsing orchestration (invokes LiteParse)
  - Text chunking and embedding generation (OpenAI, Cohere, or local models)
  - Vector upsert into PostgreSQL/pgvector
  - RAG query execution: embedding the query, vector similarity search, LLM completion with retrieved context
  - Consumes jobs from Redis for async document processing

### LiteParse (Node.js)

- **Role:** Document-to-text extraction. Handles PDF, DOCX, PPTX, HTML, and other formats.
- **Integration:** The Python worker invokes LiteParse as a subprocess via CLI. A thin wrapper script (`liteparse-cli`) accepts a file path on stdin/args and emits structured JSON (extracted text, metadata, page boundaries) on stdout.
- **Deployed** as a sidecar container or bundled into the Python worker image with Node.js runtime.

### Strapi CMS

- **Role:** Content management for platform-level content (landing pages, help articles, template definitions, knowledge-base schemas).
- **Responsibilities:**
  - Admin UI for non-developer content operations
  - REST/GraphQL API consumed by the Vue.js frontend and optionally by the Go API server
  - Manages its own PostgreSQL schema (separate logical database or schema prefix)

### Keycloak + reavencloak SPI

- **Role:** Identity provider and authorization server.
- **Responsibilities:**
  - User registration, login, MFA, password reset
  - OAuth 2.0 / OIDC token issuance (access + refresh tokens)
  - Multi-tenant realm or single-realm with org-scoped roles/groups
  - **reavencloak SPI:** Custom Keycloak Service Provider that injects tenant-specific claims (`org_id`, `org_role`) into tokens, enforces org-level invitation flows, and synchronizes user records to the Go API server's PostgreSQL via event listeners

### PostgreSQL + pgvector + ParadeDB

- **Role:** Single database engine serving relational, vector, and full-text search workloads.
- **Schema isolation:** Row-level tenant isolation using `org_id` on every tenant-scoped table; enforced via RLS policies.
- **pgvector:** Stores document chunk embeddings; supports IVFFlat or HNSW indexes for ANN queries.
- **ParadeDB:** BM25 full-text search extension; enables hybrid retrieval (vector similarity + keyword relevance) for RAG.

### Redis

- **Role:** Job queue and ephemeral cache.
- **Responsibilities:**
  - Async job queue for document processing pipeline (upload → parse → chunk → embed → index)
  - Rate limiting / request deduplication
  - Optional session or response caching

## 4. Go ↔ Python gRPC Interface

### Proto Definition (summary)

```protobuf
service AIWorker {
  rpc ParseAndEmbed (ParseRequest)      returns (ParseResponse);        // sync, small docs
  rpc QueryRAG      (RAGRequest)        returns (stream RAGChunk);      // server-streaming
  rpc GetEmbedding  (EmbeddingRequest)  returns (EmbeddingResponse);    // utility
}

message ParseRequest {
  string document_id = 1;
  string file_path   = 2;
  string org_id      = 3;
}

message RAGRequest {
  string query           = 1;
  string org_id          = 2;
  repeated string kb_ids = 3;
  map<string, string> filters = 4;
}

message RAGChunk {
  string text            = 1;
  bool   is_final        = 2;
  repeated Source sources = 3;
}
```

### Interaction Model

- **Synchronous calls** (`GetEmbedding`): Go blocks on the gRPC call; used for lightweight operations.
- **Server-streaming** (`QueryRAG`): Python streams LLM tokens as `RAGChunk` messages; Go forwards each chunk to the client via SSE. This avoids buffering the full response.
- **Async via queue** (`ParseAndEmbed`): For document uploads, Go does NOT call gRPC directly. Instead, it enqueues a Redis job. The Python worker picks it up, avoiding backpressure on the API server. The gRPC `ParseAndEmbed` RPC exists for on-demand/small-doc sync processing if needed.

### Connection Management

- Go maintains a gRPC connection pool to the Python worker (persistent connections, health-checked).
- Deadline propagation: Go sets gRPC deadlines per call; Python respects cancellation.
- Retry policy: exponential backoff with jitter for transient failures, configured in the gRPC service config.

## 5. LiteParse Integration

```
┌──────────────┐    subprocess     ┌──────────────────┐
│ Python AI    │ ──────────────►   │ liteparse-cli    │
│ Worker       │   stdin: filepath │ (Node.js)        │
│              │ ◄──────────────   │                  │
│              │   stdout: JSON    │                  │
└──────────────┘                   └──────────────────┘
```

- **Invocation:** `subprocess.run(["node", "liteparse-cli.js", "--input", path, "--output", "json"], capture_output=True)`
- **Contract:** LiteParse emits a JSON object on stdout: `{ "pages": [...], "full_text": "...", "metadata": { "page_count": N, "format": "pdf" } }`. Exit code 0 on success, non-zero with error JSON on stderr on failure.
- **Performance considerations:**
  - Node.js cold start is amortized by keeping a long-running LiteParse daemon (optional optimization: HTTP sidecar on localhost, called via `httpx` from Python).
  - For large documents, LiteParse streams page-by-page via newline-delimited JSON (NDJSON) to allow the Python worker to begin chunking before parsing completes.
- **Deployment:** LiteParse is installed into the Python worker Docker image (`apt-get install nodejs` + `npm install liteparse`), keeping it co-located for minimal latency.

### Worker Behavior

- **Concurrency:** Each Python worker instance runs N consumer coroutines (configurable, default 4). Each coroutine long-polls Redis via `BRPOPLPUSH`.
- **Visibility timeout:** When a job is popped, it moves to a processing sorted set keyed by expiry timestamp. A sweeper goroutine (or Python asyncio task) periodically checks for expired jobs and re-enqueues them.
- **Idempotency:** Workers check `document_chunks` for existing entries by `document_id` before inserting; re-processing deletes stale chunks first (DELETE + INSERT in a transaction).
- **Progress tracking:** Workers update the `documents` table with `status` transitions: `pending → parsing → chunking → embedding → ready` (or `failed`). The Go API exposes `GET /api/v1/orgs/{org}/documents/{doc}/status` for polling or webhook callback.
- **Scaling:** Additional Python worker replicas join the same Redis consumer group. Redis list semantics guarantee each job is delivered to exactly one worker.
